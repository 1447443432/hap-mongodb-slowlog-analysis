为什么慢

  • 首段 $match 没有命中可用索引，直接全表扫描。
  • $or 使用两个不同字段，通常需要“每个分支各自有索引”，否则容易退化成 COLLSCAN。
  • 这条本质上是在做计数，但用了 aggregate + $group，上游扫描成本被完整放大。
  • status 是低基数字段，不建议放进索引定义。

先分别为两个 $or 分支补单字段索引，再评估是否还需要进一步改写查询。

索引创建命令

  { "632a7c14a64afd17d12d6892": 1 },
  { background: true }
)

db.getCollection("ws66b9e2332b725be4266d4a27").createIndex(
  { "67e142b4a435fc5e6a9976e7": 1 },
  { background: true }
)

  • 先补两个 $or 分支索引。
  • 把 aggregate + $group count 改成 countDocuments(...)。

    "utime": -1
  },
  "projection": {
    "status": 0,
    "sharerange": 0,
    "wsutime": 0,
    "keywords": 0,
    "discussunreads": 0,

  },
  "limit": 1
}

为什么慢

  • 当前只命中了 utime 索引。
  • MongoDB 为了满足排序，沿 utime 扫描，再逐条回表判断过滤条件。
  • limit:1 理论上应该很快，但因为索引顺序不对，导致扫描了近 8.9 万条记录。

索引建议

高置信度建议：按“等值过滤字段在前，排序字段在后”的顺序建立复合索引。

索引创建命令

use mdwsrows
db.getCollection("ws689f0afb123c994a446ac668").createIndex(
  { "68a43976123c994a446da1fe": 1, "utime": -1 },
  { background: true }
)

可执行优化路径

  • 直接先补复合索引。
  • 建完后重新跑 explain("executionStats")。
  • 如果仍扫描很大，再排查字段值类型是否一致、是否存在脏数据。

验证建议

db.getCollection("ws689f0afb123c994a446ac668").explain("executionStats").find(
  {
    "68a43976123c994a446da1fe": "115590026165698569980",
    status: 1
  }
).sort({ utime: -1 }).limit(1)

结论

第一条是“缺少 $or 分支索引”的典型全表扫描，优先补两个分支索引，再把统计写法改成 countDocuments。第二条是“已有索引但索引顺序不对”，直接补 { 等值字段, utime: -1 } 复合索引，收益会比较直接。
